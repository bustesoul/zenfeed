# 个性化推荐系统设计文档

> 状态：草稿，持续更新中

## 一、系统目标

推荐系统的目标不是单纯推"高质量文章"，而是：

**在保证内容质量底线的前提下，让用户更快看到自己真正想看的内容，并且允许用户主动控制不想看到的内容。**

系统核心由三部分组成：

| 组件 | 职责 |
|------|------|
| **标签系统** | 定义文章特征和用户偏好/排斥 |
| **评分系统** | 表达用户对具体文章的满意度 |
| **排序系统** | 综合质量、偏好、排除条件和行为反馈，决定最终展示顺序 |

---

## 二、总原则

### 2.1 显式控制高于隐式推断

以下信号优先级最高，隐式行为只能辅助修正，不能推翻：

- 用户主动打标
- 用户主动设置标签权重
- 用户主动给文章打分
- 用户主动屏蔽主题、来源、作者或某类内容

### 2.2 质量分与用户偏好分必须分开

- **内容质量分**：文章本身是否扎实、值不值得看（与具体用户无关）
- **用户匹配分**：这篇文章是否适合当前用户

排序时再组合，不允许混为一谈。

### 2.3 打标不默认等于"喜欢"

用户打标可能表达三种完全不同的意图：

- 这是我想看的内容
- 这是我不想看的内容
- 这是我希望系统识别并区别对待的内容

因此标签必须带有方向性语义，不能只存"有这个标签"。

## 三、标签系统

### 3.1 双主轴结构

**第一轴：主题标签**——描述文章讲什么

例如：`AI`、`Rust`、`数据库`、`创业`、`投资`、`开发工具`

**第二轴：内容形态标签**——描述文章怎么表达

例如：`快讯`、`深度分析`、`教程`、`观点`、`盘点`、`实战案例`

> 判断偏好时必须同时看两个轴。同样是 AI，用户可能喜欢`AI + 深度分析`，但不喜欢`AI + 快讯`。

### 3.2 标签来源三层

| 层级 | 来源 | 权重 |
|------|------|------|
| 系统标签 | 模型或规则自动生成 | 低 |
| 候选标签 | 系统给出，供用户确认或修正 | 中 |
| 用户标签 | 用户主动添加、删除、修正 | 高（覆盖系统标签）|

### 3.3 控制粒度到"主标签 + 子标签"

用户不仅可以控制大类，也可以控制更细的方向：

```
AI
AI / 工具
AI / 模型
AI / 融资
Rust / 教程
Rust / 工程实践
```

### 3.4 标签必须带动作语义

| 动作类型 | 含义 | 示例 |
|----------|------|------|
| **正向偏好** | 希望多看这类内容 | 多推 `AI / 工具`、偏好`深度分析` |
| **负向偏好** | 希望少看，但不绝对禁止 | 减少`快讯`、减少`入门教程` |
| **排除** | 明确不想看到 | 屏蔽`八卦`、屏蔽某来源/作者 |
| **识别** | 系统记住此类型，不直接表达喜恶 | `标题党`、`信息重复`、`偏营销`、`适合碎片阅读` |

> **识别标签**用于后续排序解释和行为学习，不应默认算作正向偏好。

### 3.5 内容标签 vs 控制标签必须分开存储

| 类型 | 描述 | 示例 |
|------|------|------|
| **内容标签** | 描述文章本身是什么 | `AI / 工具`、`Rust / 教程`、`深度分析` |
| **控制标签** | 描述用户希望系统怎么处理 | `多推`、`少推`、`屏蔽`、`标题党风险`、`值得收藏` |

用户标签存储格式示例：

```json
{ "tag": "AI / 工具",  "action": "boost" }
{ "tag": "快讯",       "action": "demote" }
{ "tag": "标题党",     "action": "flag" }
{ "tag": "某作者",     "action": "block" }
```

### 3.6 后端标签编码方案：`article_analysis` JSON label

现有 `Labels` 模型是单值 KV（同 key 覆盖），且 rewrite 每条规则只输出一个 label。  
为了用**一次 LLM 调用**拿到所有结构化元数据，采用单一 JSON blob label 方案：

**Label 名**：`article_analysis`  
**生成方式**：独立 rewrite 规则，使用便宜模型（非 gpt-5.4）  
**内容格式**：

```json
{
  "base_quality_score": 8,
  "primary_topic": "AI/工具",
  "format": "深度分析",
  "topics_json": ["AI/工具", "开发效率"]
}
```

| 字段 | 用途 |
|------|------|
| `base_quality_score` | 重排底盘分，0-10 整数 |
| `primary_topic` | 主标签，供重排和 UI 展示使用 |
| `format` | 内容形态，供重排和 UI 展示使用 |
| `topics_json` | 多主题列表，供反馈面板和重排使用 |

**现有 inverted index 影响**：无。现有 filter/group_by 继续基于原有 labels 工作，不依赖 `article_analysis`。Phase 3+ 若需要 `primary_topic` 进入可检索索引，可额外加一条规则单独输出该 label。

> **不要**将 `topics_json` 直接存为独立 label——现有过滤系统把 label value 当单值精确匹配，存 JSON 数组字符串会导致无法识别其中的独立标签。

## 四、评分系统

### 4.1 十分制，6分及格线

| 区间 | 语义 |
|------|------|
| 1–3 | 强负向（明显不满意）|
| 4–5 | 弱负向（一般，不太值）|
| 6   | 中性及格（可看）|
| 7–8 | 正向（不错，值得继续推荐相似内容）|
| 9–10 | 强正向（强认可，明显放大相似内容和相关标签）|

> 评分按**分段**使用，不按线性使用。8分不是4分的两倍。

### 4.2 低分影响传播（保守）

| 分段 | 当前文章 | 相似内容 | 子标签簇 | 来源/作者 |
|------|----------|----------|----------|-----------|
| 1–3  | 强降 | 降高相似 | 轻降 | 仅重复出现时 |
| 4–5  | 降 | 降相似 | 微降 | 默认不影响 |

> 低分传播必须比高分更保守，避免偶然误伤。

### 4.3 高分奖励三层对象

优先级：**标签簇 > 相似内容 > 来源/作者**

高分最能说明用户喜欢这一类内容，其次是表达形式，最后才是来源或作者。

### 4.4 评分支持附带理由

同样低分，原因可能完全不同：

```
质量低 / 不对口 / 太浅 / 太水 / 重复信息 / 太长 / 标题党 / 立场不喜欢
```

附带理由会显著提高后续规则质量。

## 五、显式控制

### 5.1 标签控制支持多档

基础三档：**多推 / 少推 / 屏蔽**

扩展六档：强多推 / 多推 / 默认 / 少推 / 强少推 / 屏蔽

### 5.2 用户主动打标立即生效

不应等待长期行为验证。用户明确表达的控制，系统立即进入排序逻辑。

### 5.3 排除标签强过滤

对用户明确设置的屏蔽（主题、子标签、内容形态、来源、作者），默认不参与普通排序，直接过滤。

### 5.4 控制面板可查看、可撤销、可重置

至少包含：

- 当前多推标签
- 当前少推标签
- 当前屏蔽项
- 最近主动打的标签
- 单项删除/重置
- 一键恢复默认

## 六、行为反馈（辅助层）

> 这些规则是辅助层，不是主系统核心。

### 6.1 已读无正反馈 → 中性

用户点开并完成基本阅读，但没有打分/收藏/分享，默认视为中性信号，说明文章"可消费"，但不能推断喜好。

### 6.2 连续重复才做轻微负向修正

阈值建议：连续 3–4 次"已读但无正反馈"，才触发轻微修正。

### 6.3 "同类内容"按主题 + 内容形态联合定义

行为修正粒度至少到：`主题标签 + 内容形态`（例如：`AI + 快讯`、`Rust + 教程`），避免把"对某种形式无感"误判为"对整个主题无感"。

### 6.4 未点击 > 已读无反馈（负面程度）

- 未点击：主题入口就不成立，负面程度更高
- 已读无反馈：至少说明文章有一定消费价值

### 6.5 行为负向修正同时作用于排序和曝光频次

不是只改排序，也不是只控频，两者同时轻微调整。

### 6.6 行为负向修正慢速衰减

应慢速衰减（数周量级），不应几天内消失，因为它更像稳定偏好线索。

### 6.7 后续强正反馈可部分修正但不立即反转

## 七、基础质量分（base_quality_score）

### 7.1 base_quality_score 与用户无关，独立存在

**Phase 1 范围：仅 LLM 内容质量评估**（不依赖任何用户行为数据）

`base_quality_score` 由独立 rewrite 规则生成，存储在 `article_analysis` JSON label 中，与摘要生成规则分开，使用便宜模型：

```yaml
llms:
  - name: cheap          # 新增便宜模型，专用于结构化提取
    provider: ...
    model: ...           # 如 gpt-4.1-mini、qwen3 等

rewrites:
  # 现有：摘要生成（gpt-5.4，质量优先，保持不变）
  - transform:
      to_text:
        prompt: "{{ .summary_html_snippet_for_small_model }} Respond in Chinese"
    label: summary_html_snippet

  # 新增：结构化分析（便宜模型，一次调用拿全部元数据）
  - transform:
      to_text:
        llm: cheap
        prompt: |
          分析这篇文章，仅返回JSON，不要其他内容：
          {"base_quality_score":7,"primary_topic":"AI/工具","format":"深度分析","topics_json":["AI/工具"]}
    label: article_analysis
```

构成因素（Phase 1）：

- LLM 对文章内容扎实度、信息密度的评分
- 标题与正文一致性（由 LLM rewrite pipeline 判断）
- 内容重复度（去重时已处理）

> 命名为 `base_quality_score`，不叫"全局质量分"——现阶段只有 LLM 打的分，"全局"语义虚高。

以下因素依赖尚不存在的行为数据（已读状态目前仅在 localStorage），推迟到 Phase 3：

- 全局读完率 / 快速退出率
- 全局低分率
- 来源 / 作者历史质量趋势

### 7.2 质量底线优先

即使用户偏好某标签，低于质量底线的内容也不进入高位排序，只进入探索位或直接过滤。

## 八、排序系统

### 8.1 三阶段排序

**第一阶段：召回（扩量）**

召回时必须用扩大的 `recall_limit`，而不是直接用展示 `limit`——否则个性化重排只能在已被截断的结果里洗牌，无法把本该进来的文章救回来：

```
recall_limit = max(limit × 5, 50)
```

召回维度：

- 主题标签 / 子标签 / 内容形态
- 相似文章
- 来源/作者
- 少量探索内容

**第二阶段：过滤**

- 用户屏蔽内容
- 明显低于质量底线的内容
- 高重复 / 过度相似内容

**第三阶段：重排（API 层）**

综合计算最终得分，**个性化重排在 API 层做，不在 rewrite pipeline**（rewrite 只在写入时跑一次，分数写入后冻结，跟不上用户后续反馈）：

```
final_score =
  base_quality_score     // 内容基础质量分（底盘，LLM 评估）
  × tag_control_weight   // 用户标签控制（强乘子）
  + rating_history       // 用户直接打分历史
  + format_preference    // 内容形态偏好
  + freshness            // 新鲜度
  - diversity_penalty    // 去重与多样性修正
  + behavior_signal      // 行为反馈修正（辅助，衰减）
```

重排完成后，截取最终 `limit` 条返回。

### 8.2 显式标签权重是强乘子，但建立在质量底线上

先通过质量底线，再被显式偏好强拉升。这样既保留控制感，也不会把低质量内容顶到前面。

### 8.3 推荐目标：命中效率优先

- 强响应显式控制
- 高分内容快速放大
- 低分内容快速收缩
- 排除条件立即生效

## 九、标签与评分的联动

### 9.1 分工明确

- **标签**：告诉系统"推什么 / 少推什么 / 不推什么"（方向）
- **评分**：告诉系统"这一篇做得好不好，值不值得继续放大"（强度）

### 9.2 用户打标可能是"排除"，系统必须识别

以下标签应默认进入少推/排除通道，而不是正向偏好模型：

```
营销号 / 标题党 / 噪音高 / 不想看 / 重复
```

## 十、文章归档（已确认）

### 两类文章，两种命运

| 类型 | 条件 | 存储行为 |
|------|------|---------|
| **关键文章** | 用户主动打分或打标签 | 永久保存，豁免 retention 清理 |
| **普通文章** | 未打分、未打标 | 走现有 retention 周期（默认 8 天）自动清理 |

### 个人知识库

独立于普通 feed 流存储，支持：

- 按分数筛选（≥8分精华）
- 按标签筛选
- 按来源、分类、时间过滤
- 语义搜索（基于已有 embedding 能力）

### Archive 数据结构（草稿）

```json
{
  "feed_id": "原始 feed 的 id",
  "archived_at": "2026-04-21T11:00:00Z",
  "original_labels": { "title": "...", "source": "...", "content": "..." },
  "user_feedback": {
    "score": 8,
    "score_reason": "深度好，信息密度高",
    "tags": [
      { "tag": "AI / 工具", "action": "boost" }
    ],
    "note": "用户备注（可选）"
  }
}
```

## 十一、架构设计

> **单用户范围约定**：本系统面向单用户自托管场景。现有认证是单组环境变量凭证，不存在多用户实体。所有 KV key 均不使用 `user:*` 前缀，避免误导。

### stable feed_id 暴露

后端已有稳定 uint64 ID（来自抓取链路），但当前 `model.Feed.ID` 无 json tag，前端拿不到，只能用 label 哈希作为临时 ID（换 label 就变了，不稳定）。

**必须修改**：将 `Feed.ID` 以**字符串形式**暴露给前端（不能裸出 uint64 数值，JS 对 > 2⁵³ 的整数有精度丢失问题）。

前端改成统一用字符串 id，`getFeedItemId()` 替换为后端返回的稳定 id。

### KV 存储设计（NutsDB）

当前 KV 是 NutsDB（非 bbolt），只有 Get/Set，无 Delete/List/Scan。采用两类 key，Profile 用 read-modify-write 整 blob 模式，无需 KV 扩展：

| Key | 内容 | 说明 |
|-----|------|------|
| `profile:global` | 聚合偏好 blob（标签权重 + 控制规则）| 小，频繁读写 |
| `feedback:{feed_id}` | 单篇原始反馈 JSON | 每篇一条，不随 feed 清理 |
| `read:{feed_id}` | 已读标记（时间戳）| 替代 localStorage |
| `archive:{feed_id}` | 归档内容 + feedback 快照 | 永久保留 |

`profile:global` 只存聚合结果（当前偏好权重），不存全部反馈历史——避免越存越大。反馈历史由 `feedback:{feed_id}` 分散存储。

### 新增组件

| 组件 | 路径 | 说明 |
|------|------|------|
| `article_analysis` rewrite 规则 | `config.yaml` | 便宜模型生成结构化元数据 JSON label |
| Feedback API | `pkg/api/http/` JSON-RPC 风格 | 接收用户评分、标签（含动作语义）|
| Archive API | `pkg/api/http/` JSON-RPC 风格 | 归档文章，豁免 retention |
| Preference 读取 API | `pkg/api/http/` JSON-RPC 风格 | 返回当前 profile:global |
| 个性化重排 | `pkg/api/http/` 查询层 | 召回扩量 → 重排 → 截取 |
| 前端反馈 UI | zenfeed-web，feed 卡片 | 10分 + 标签（含动作）+ 理由 |
| 用户控制面板 | zenfeed-web | 查看/编辑/重置标签控制规则 |

### API 设计（JSON-RPC 风格，与现有 /query /write 保持一致）

```
POST /api/feedback
Body:
{
  "feed_id": "1234567890",      // 字符串形式的 stable id
  "score": 8,
  "score_reason": "深度好",
  "tags": [
    { "tag": "AI / 工具", "action": "boost" },
    { "tag": "快讯",      "action": "demote" }
  ],
  "archive": true
}

GET /api/preference
Response:
{
  "tag_controls": [
    { "tag": "AI / 工具", "action": "boost" },
    { "tag": "快讯",      "action": "demote" },
    { "tag": "某来源",    "action": "block" }
  ],
  "feedback_count": 42,
  "last_updated": "2026-04-21T11:00:00Z"
}
```

### personal_score 计算位置

**不在 rewrite pipeline**（rewrite 在写入时跑一次，分数冻结，无法响应后续反馈）。

个性化重排在 **API 查询层**完成：

```
recall_limit = max(limit × 5, 50)
// 1. 用 recall_limit 召回候选集
// 2. 对每篇计算 personal_score，并记录匹配的偏好规则：
personal_score =
  base_quality_score                                          // 质量底盘（LLM 分）
  × tag_control_multiplier                                   // 标签控制强乘子
  + α * cosine_similarity(article_embedding, interest_vector)
  + β * rating_history_signal
  + (1 - α - β) * format_preference_signal
// 3. 按 personal_score 降序排列，截取 limit 条返回

// 冷启动（无反馈）时 α=β=0，只用 base_quality_score
// 每条结果附带 matched_preferences []string，供前端渲染推荐原因标签
```

## 十二、前端设计方案

### 现状

| 现有能力 | 问题 |
|---------|------|
| 已读状态存 localStorage | 换设备丢失，无法跨端同步 |
| 右键菜单：仅"标记已读" | 无打分/打标入口 |
| 主页 Tab：Past / Notifications / Advanced | 无知识库入口 |

### 改动一：feed 卡片右键菜单（已确认）

保持简洁，只有两项：

```
┌─────────────────────┐
│ ✓  标记已读          │  ← 保留，快捷操作
│ ──────────────────  │
│ 📋 反馈此文章...     │  ← 新增，打开反馈面板
└─────────────────────┘
```

### 改动二：反馈面板（Slide-over，已确认）

点击"反馈此文章"后右侧滑出，所有操作集中在一个面板，**全部带默认值，方便快速确认提交**：

```
┌──────────────────────────────┐
│  反馈此文章              ✕   │
```

### 改动四：推荐系统可见性设计

#### 4.1 学习进度指示（主页顶部，仅冷启动阶段显示）

反馈数 < 5 条时，主页 feed 列表顶部显示引导条：

```
┌─────────────────────────────────────────────────────────┐
│  🧠 个性化学习中  ██░░░░░░░░  2/5 篇  给更多文章反馈以激活排序  │
└─────────────────────────────────────────────────────────┘
```

反馈数 ≥ 5 后该条消失，不再打扰。

#### 4.2 里程碑 Toast（一次性触发）

| 触发条件 | Toast 内容 |
|----------|-----------|
| 第 1 条反馈提交 | `🎉 偏好学习已启动，再给 4 篇反馈可激活个性化排序` |
| 第 5 条反馈提交 | `✨ 个性化排序已激活！已发现你偏好「AI/工具 · 深度分析」` |
| 第 20 条反馈提交 | `📈 偏好画像已成熟，推荐准确度持续提升中` |

每个里程碑只触发一次，记录在 localStorage，不重复弹出。

#### 4.3 feed 卡片推荐原因标签

个性化排序激活后（反馈 ≥ 5），每张 feed 卡片左下角显示小标签，说明为什么被提升：

```
┌──────────────────────────────────────┐
│  GPT-5 工具链实战                     │
│  Github 热榜 · 2026-04-20             │
│  ↑ AI/工具  ↑ 深度分析                │  ← 匹配了哪个偏好
└──────────────────────────────────────┘
```

被压低的文章（匹配了少推规则）不显示标签，只是排序靠后，不做视觉强调（避免负面感）。

后端重排时在每条结果附上 `matched_preferences: ["AI/工具", "深度分析"]`，前端直接渲染。

#### 4.4 偏好面板加"本周变化"diff

在偏好画像右侧加本周增量，让用户看到系统在持续学习：

```
┌──────────────────────────────────────────────────────┐
│  偏好画像                    基于 42 篇反馈学习        │
│                                                      │
│  正向：                                              │
│  AI/工具  ████████ 0.9  (+0.2 本周)                  │
│  Rust     ██████   0.6  (+0.1 本周)                  │
│                                                      │
│  负向：                                              │
│  快讯     ████████-0.8  (-0.3 本周)                  │
└──────────────────────────────────────────────────────┘
```

diff 数据存在 `profile:global` blob 里（每次更新时保存上次快照）。

#### 4.5 完整的用户感知弧线

```
Day 1，第 1 篇反馈
  → Toast：偏好学习启动，进度条出现

Day 1，第 5 篇反馈
  → Toast：个性化排序激活
  → feed 卡片开始出现「↑ AI/工具」推荐原因标签
  → 进度条消失

Week 1，累计 ~20 篇
  → Toast：偏好画像已成熟
  → 偏好面板出现明显权重分布 + 本周 diff

持续使用
  → 知识库积累文章，可回看兴趣演变
  → 偏好面板 diff 持续更新，系统"活着"的感觉清晰
```
